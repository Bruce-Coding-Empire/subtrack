Since TS 4.1, `paths` resolves relative to the tsconfig file's own location when `baseUrl` is unset (path values must then start with `./`) — this matches the pattern `apps/web/tsconfig.json` already uses. Verified empirically (not assumed) that the Nest CLI's own compiler — used by both `nest build` and `nest start --watch` — already rewrites `@/...` imports to correct relative `require()` paths in the emitted JS, so no extra runtime tooling (`tsc-alias`, `tsconfig-paths/register`) is needed; a first attempt to add `tsc-alias` as a belt-and-suspenders measure was reverted once testing showed it was redundant. Plain `tsc` (not routed through `nest build`) does *not* do this rewrite — don't reach for raw `tsc` for anything that needs `@/` imports in this app.

- **04 Auth — Web UI + Wiring.** `apps/web` runs on `localhost:3000`, `apps/api` on `localhost:3001` — genuinely cross-origin, not just cross-port-same-app. `main.ts` had no `enableCors()` call at all before this feature (feature 03 was only ever hit via curl/Swagger, never a browser), so the first browser `fetch()` from a web page would have been silently blocked. Added `app.enableCors({ origin: WEB_APP_URL, credentials: true })` in `main.ts` — `credentials: true` is required for the browser to accept/send the httpOnly refresh cookie cross-origin. New env var `WEB_APP_URL` (`http://localhost:3000` in dev) added to `apps/api/.env(.example)`, not previously in `code-standards.md`'s env table since nothing needed it until a browser was in the picture.
- **Middleware is `proxy.ts` in Next.js 16, not `middleware.ts`** (confirmed via Context7 docs for `next@16.2.9`, matches the installed `16.2.10` — see https://nextjs.org/docs/messages/middleware-to-proxy). File lives at `apps/web/proxy.ts` (project root, alongside `app/`), default-exports a `proxy()` function, same `NextRequest`/`NextResponse`/`config.matcher` API as the old `middleware.ts`.
- **`proxy.ts` cannot read the API's refresh-token cookie** — that cookie is `httpOnly` and scoped to the API's own origin (`localhost:3001`); a request to the web app's origin (`localhost:3000`) never carries it, so the Next.js server has no accessible signal of auth state at all (the access token is browser-memory-only, per `build-plan.md`, and is lost on every full navigation/reload anyway). Resolved by having `lib/auth.ts` set a second, plain (non-`httpOnly`) marker cookie — `subtrack_session`, `apps/web/lib/constants.ts` — on the web app's own origin whenever a session is established (login/register success, or a successful silent refresh), and clearing it on logout/refresh-failure. `proxy.ts` reads only this marker to decide whether to redirect `/dashboard`, `/subscriptions`, `/settings` → `/login` (no marker) or `/login`, `/register` → `/dashboard` (marker present). This is optimistic/UX-only routing, matching the "Optimistic Authentication" pattern Next's own docs recommend for this exact situation — real authorization still happens at the API on every request via the Bearer access token; the marker proves nothing to the API and is never trusted as a security boundary.
- Lazy/self-healing access-token refresh implemented exactly as `code-standards.md`'s `apiFetch` pattern describes: `apps/web/lib/api-client.ts`'s `apiFetch()` attaches the in-memory access token if present, and on a `401` (excluding `/auth/*` paths, to avoid recursion and to let login/register surface real "invalid credentials" errors instead of triggering a refresh attempt) calls `refreshAccessToken()` once, retries, then clears the session and hard-redirects to `/login` if the retry also fails. No separate "bootstrap on app load" step exists yet — the first protected data call after a reload naturally 401s (empty in-memory token) and self-heals via the refresh cookie. Flag for feature 05+: Server Component data fetching against protected endpoints isn't resolved — the access token lives in browser memory only, so an SSR `fetch` from a Server Component has no token to attach. Likely resolution is client-side (`"use client"`) data fetching for protected pages, but not decided now since feature 04 has no protected page content to fetch.
