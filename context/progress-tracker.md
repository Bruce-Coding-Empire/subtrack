- **`proxy.ts` cannot read the API's refresh-token cookie** — that cookie is `httpOnly` and scoped to the API's own origin (`localhost:3001`); a request to the web app's origin (`localhost:3000`) never carries it, so the Next.js server has no accessible signal of auth state at all (the access token is browser-memory-only, per `build-plan.md`, and is lost on every full navigation/reload anyway). This is optimistic/UX-only routing, matching the "Optimistic Authentication" pattern Next's own docs recommend for this exact situation — real authorization still happens at the API on every request via the Bearer access token; the marker proves nothing to the API and is never trusted as a security boundary.
